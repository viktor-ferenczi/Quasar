# Grid Viewer

Quasar includes a first-pass browser viewer for live grid and asteroid voxel entities listed on the Entities page.

## How to open it

1. Open **Entities**.
2. Select a connected server.
3. Refresh the entity list.
4. Click the eye icon beside a grid or asteroid voxel entity.

The viewer opens grid rows as `/viewer/entity?agentId=...&entityId=...&voxels=1&context=1` and asteroid rows as `/viewer/entity?agentId=...&entityId=...&voxels=1`. It requests a scene snapshot from `/api/viewer/entities/{agentId}/{entityId}/scene`. Direct viewer URLs without a `voxels` parameter keep voxel data support disabled; direct grid URLs without a `context` parameter keep context mode disabled.

## Asset Boundary

Quasar does not serve Space Engineers assets to the browser.

The viewer endpoint returns scene metadata and, when explicitly requested with `voxels=1`, bounded voxel terrain content/material samples intersecting the selected grid bounds or the selected asteroid's live storage:

- grid identity, transform, size, bounds, and static/dynamic state
- optional context-grid identity, transforms, clipped bounds, and block placement for loaded grids near the selected grid
- scene lighting metadata, including the current in-game sun direction and captured block and runtime-subpart light sources
- loaded voxel body metadata and world bounds for planets, asteroids, moons, and voxel maps
- optional voxel data chunks bounded to the viewer floor-grid footprint for grids, or split across the selected asteroid storage for standalone asteroid scenes; the browser procedurally meshes these chunks from the scalar field
- voxel material definition metadata for sampled material indexes, including logical voxel texture paths
- block definition IDs and block placement
- block cell coordinates, orientation, color mask, skin ID, build state, integrity, and accumulated damage
- logistics metadata for every rendered grid: conveyor/inventory endpoint nodes, physical conveyor links, connected-system IDs, working state, and small-conveyor restriction flags
- logical model paths for block definitions, current block models, generated cube parts, and runtime subparts
- logical texture paths only when they are available as metadata
- LCD text surface metadata, including screen material names, selected image IDs, logical image/sprite texture paths, text content, colors, font settings, and sprite draw-layer metadata
- non-fatal warnings

The endpoint must not return:

- raw `.mwm` bytes
- raw texture bytes
- server-rendered LCD texture bytes
- raw extracted game model geometry, UVs, or material texture data
- a generic asset download API

Voxel data payloads are sampled by the running server from live voxel storage and are limited by conservative chunk and byte budgets. They do not include Space Engineers `.mwm`, `.dds`, `.png`, or other asset files.

## Local Content Folder

The browser asks the user to select their local Space Engineers `Content` folder. The folder should contain `Data`, `Models`, and `Textures` directories.

On Chromium browsers, the viewer uses the File System Access API (`showDirectoryPicker`) and keeps the selected folder handle in browser storage when supported, so the viewer can reuse it on later visits after permission is granted. On Firefox, which does not support `showDirectoryPicker`, the viewer falls back to a browser folder input (`webkitdirectory`) and builds an in-memory view of the selected files. Firefox users must select the Content folder again after a page reload.

The viewer resolves logical model, texture, and SBC data paths case-insensitively where the selected browser file API permits it.

## Local Mods Folder

Modded grids can also use assets from a user-selected global Space Engineers `Mods` folder. Select the folder that directly contains mod directories, `.sbm` archives, or legacy `*_legacy.bin` packages, for example `%AppData%/SpaceEngineers/Mods` on typical Windows client installs. Steam Workshop layouts such as `steamapps/workshop/content/244850`, where workshop items are directories named by published file ID, are also supported. The viewer stores this folder handle independently from the local `Content` folder when the browser supports persistent directory handles.

Quasar still sends metadata only. The agent includes a metadata-only list of active scene mods and root hints on model assets, but it does not send server-local mod paths, model bytes, texture bytes, extracted geometry, or archive contents.

When a model or texture has a mod root hint, the browser resolves it from the matching active mod under the selected global Mods folder first, then falls back to the selected vanilla `Content` folder. Logical paths that already start with `Mods/<mod-name>/...` are resolved directly under that mod name. For older or ambiguous snapshots marked as mod content without a root hint, the browser may try active scene mods in scene order after the Content fallback.

Unpacked mod directories, `.sbm` archives, and legacy Steam Workshop `*_legacy.bin` packages are supported. Legacy packages are zip-backed archives stored inside Workshop item directories; the viewer opens them locally through the same zip reader only when an asset from that mod is requested, strips the package filename from server-local legacy paths such as `*_legacy.bin/Models/...`, and indexes archive entries lazily and case-insensitively. Asset bytes stay on the user's machine.

Folder selection is optional and non-fatal:

- With only `Content` selected, vanilla assets render as before and modded assets may fall back to proxies.
- With only `Mods` selected, modded assets may resolve, but vanilla fallback assets can be missing.
- With neither folder selected, models render as proxies and textures use generated fallbacks.
- If an active scene mod is not found under the selected Mods folder, the viewer logs a warning once for that mod and keeps rendering.

## Current Rendering Behavior

The browser always renders the selected grid in a relative view. The grid's in-game world transform is still used, but the view is re-centered on the grid and rotated into the grid's local frame so large world coordinates do not reduce browser-side precision. Context grids, voxel body proxies, and the sun direction are transformed through the same relative frame.

The floor grid is scaled to the displayed grid bounds, or to the full context volume footprint when context mode is enabled, padded by two major squares, and uses Space Engineers cube-size semantics: 0.5 m sub-squares are always shown, while major lines use the selected grid size (`0.5 m` for small grids and `2.5 m` for large grids). Context mode always uses large-grid major spacing (`2.5 m`) even when the selected grid is small, because the context volume is clamped and sampled at large-grid scale. Context mode only changes the floor grid footprint; the floor height remains anchored below the selected grid so its vertical offset relative to the target grid does not change. Because the viewer centers the selected grid, floor-grid major lines are offset independently on X and Z according to the active floor-grid footprint parity so they align with block boundaries for both even and odd grid dimensions. World-axis lines are not highlighted separately.

Context mode is controlled by the `context` URL parameter and the `Context mode` checkbox. `context=1`, `context=true`, `context=yes`, and an empty `context` value request context; `context=0`, `context=false`, and `context=no` disable it. When enabled for a grid, the agent builds the context volume from the selected grid's local AABB by clamping each half-extent to at least one large-grid cube (`2.5 m`) and then doubling those half-extents. The agent sends both an enclosing world AABB for server-side voxel queries and a selected-grid-relative AABB for browser floor-grid and voxel clipping, so rotated grids do not inflate the viewer's context footprint by transforming a world-axis AABB back into grid space. Loaded grids first pass the enclosing world AABB broad phase, then their oriented grid bounds must intersect the oriented local context volume. Surrounding grids are clipped at block selection time with oriented block bounds against the same context volume, so very large neighboring grids do not transmit distant blocks. Boundary blocks may protrude because mesh triangle clipping is intentionally not performed in this pass.

The viewer parses locally resolved `.mwm` files in the browser and renders mesh geometry for block models, generated cube-part models, and runtime subpart models. Current parsing covers the render mesh tags needed for static geometry (`Vertices`, `Normals`, `TexCoords0`, `MeshParts`, `PatternScale`) and follows `GeometryDataAsset` indirection used by stub MWMs. MWM material parsing also preserves `GLASS` material metadata (`GlassCW`, `GlassCCW`, and smooth-normal flag), which the renderer resolves through the local `Data/TransparentMaterials.sbc` file to the same transparent glass texture/color records used by Space Engineers. MWM triangle indices are converted from Space Engineers' clockwise Direct3D front-face winding to Three.js/WebGL counter-clockwise winding so sun lighting uses the same face orientation as the visible sun line. Regular block models use the exported full local block matrix directly; `ModelOffset` is only applied for legacy snapshots that lack that matrix, avoiding double-offsets on blocks such as hangar doors. Model file resolution/parsing runs with bounded parallelism instead of resolving every listed model one at a time. The viewer first displays temporary proxy boxes so the grid is visible while model files are being resolved and parsed, then progressively rebuilds the model layer on animation frames as parsed assets become available. Scene construction no longer waits for all referenced texture paths to resolve; model geometry appears with generated fallback materials first, then selected material textures fill in progressively. While real models are still loading, unresolved blocks are drawn as instanced proxy batches grouped by opacity instead of one standalone proxy object per block; the proxy solids keep each block's paint color and their outlines are rendered as batched cuboid edge lines.

The `Show logistics` control overlays the rendered grids' conveyor/logistics topology without rebuilding the base model layer. The agent sends metadata only: endpoint/inventory node IDs, owning grid IDs, block roles, conveyor port counts, physical conveyor line endpoints, conveyor-route path points, line type, working state, and connected component IDs. The browser draws connected conveyor links in one neutral colour, draws small-conveyor-restricted links as dashed lines, and colours non-working links orange. Non-working links are excluded from connected-component detection, so an offline bridge does not merge the systems on either side for hover highlighting; the browser splits the orange offline path at its midpoint so each half highlights with the system on that side. Conveyor tracers follow the actual conveyor path through intermediate conveyor blocks rather than drawing straight chords between endpoints. Conveyor paths that extend from a node but do not reach another endpoint are drawn red and reported as open conveyor paths. Logistics overlays are slightly dimmed by default; hovering a line or conveyor-connected logistics block brightens its system, while hovering a logistics block with no conveyor ports brightens every conveyor-less logistics block on that grid. Conveyor tracers and logistics block masks render through walls so buried routing remains visible. Logistics blocks get translucent role-colour masks applied to their loaded block models instead of bounding cubes: storage is blue, production is orange, power is red/pink, gas is green/teal, weapons/tools are violet, connectors/sorters are yellow, and plain conveyor blocks use muted gray. The stats panel reports logistics systems, nodes, edges, small conveyor links, and open conveyor paths across the visible scene. Hover readout shows logistics role or edge line type, system ID, and working state.

The `Show damaged` control is currently mutually exclusive with `Show logistics`. It reuses the same through-wall model-mask overlay technique to draw blocks selected by the projector block's unfinished/damaged marker predicate: `BuildLevel` is greater than zero and less than one, `AccumulatedDamage` is greater than zero, or current damage (`BuildLevel * MaxIntegrity - Integrity`) is greater than zero. The aura uses the largest of missing build integrity, accumulated damage, and current damage as its intensity, so lightly affected blocks are orange-red and translucent, while heavily affected blocks become more saturated red and more opaque. When voxel data is present, the agent also sends voxel damage chunks with a per-sample diff mask built by comparing current storage against the voxel body's data provider, which is the original/generated terrain source used by unmodified storage leaves. The browser only emits aura polygons touching changed samples, so the overlay follows the actual deformation instead of highlighting the whole voxel body or a coarse modified storage leaf. Voxel deformation aura visibility follows both `Show damaged` and `Show voxels`; turning voxel rendering off hides the voxel deformation overlay while leaving damaged block masks available. Hovering a highlighted block brightens all highlighted blocks on that block's grid and dims highlighted blocks on other visible grids; hovering a voxel deformation brightens every highlighted modified chunk from the same voxel body. The stats panel reports highlighted damaged/unfinished blocks and highlighted voxel damage chunks in the scene.

## Viewer Load Performance

Recent browser diagnostics showed that the remaining slow path is mostly browser-side local `Content` folder work and repeated large proxy-scene rebuilds, not raw MWM parsing.

The production viewer now uses a wider but still bounded set of local filesystem queues: path lookups run with up to `32` concurrent operations, and deferred `getFile()` metadata snapshots run with up to `16`. Model resolution/parsing is capped at `48` workers so the browser stays busy without piling hundreds of model tasks behind narrower filesystem queues.

Progressive model-layer rebuilds are also throttled adaptively. Smaller scenes rebuild sooner so visible geometry appears quickly, while larger scenes wait for more resolved models or a longer maximum delay before rebuilding. This reduces repeated near-identical rebuilds on large grids while preserving the immediate proxy-first load path.

Armor and other generated cube-part models use the game-provided `PatternOffset` metadata from `MyCubeGrid.GetCubeParts(...)`. The browser applies the MWM `PatternScale` to raw model UVs and then applies the cube-part pattern offset using the same final UV expression as the Space Engineers vertex shader. This preserves asymmetric corner and slope triangle atlas layouts such as `LargeBlockArmorCorner` and `LargeBlockArmorSlope2Tip`. Regular block models and runtime subparts do not receive cube-part pattern offsets; all MWM model UVs stay in Space Engineers shader orientation, and loaded textures opt out of Three.js' default vertical flip so panel decals and control-panel textures map the same way as in game.

Model material textures are also resolved from the parsed model's source root first, then from the selected local `Content` folder. Browser-native image files are loaded directly, and DDS material textures are parsed in the browser for common Space Engineers compressed formats including DXT1/DXT3/DXT5, BC4, BC5, and DX10 BC7. Loaded textures use repeat wrapping without Three.js' default `flipY`, matching the Direct3D-oriented UVs stored in Space Engineers MWM assets. DDS upload still depends on the user's browser/GPU exposing the matching WebGL compressed-texture extension, such as `WEBGL_compressed_texture_s3tc`, `EXT_texture_compression_rgtc`, or `EXT_texture_compression_bptc`. Texture color-space classification prefers the material slot name and otherwise uses filename tokens such as `_ng`, `_add`, `_alphamask`, and `_orm`; directory or asset words like `Cubes` or `Armor` do not turn a color-metal texture into a data map. Compressed DDS uploads also preserve an explicit sRGB DX10 format tag even for slots otherwise classified as data, matching Space Engineers content where some normal-gloss files store linear values in sRGB-encoded BC7 blocks. Transparent material `GlossTexture` maps are treated like the Space Engineers glass shader treats them: the alpha channel drives gloss/roughness, but RGB does not perturb the glass normal. Texture work uses separate limits for path resolution, metadata reads, file reads/DDS parsing, and WebGL upload so filesystem lookups can run wider while upload memory pressure stays bounded. Duplicate logical texture loads and extensionless texture candidate resolutions are coalesced before file metadata is known, then completed textures still use the source root, local file size, and `lastModified` cache key for invalidation.

LCD text surfaces are treated the same way: the agent sends only text-surface metadata and logical texture paths for selected LCD images or programmable-block sprite layers. The agent discovers standalone LCD panels through their active panel component and discovers integrated screens through Space Engineers' `IMyMultiTextPanelComponentOwner` interface plus the block definition `ScreenAreas`, covering cockpit/control-seat displays and other screen-capable blocks listed in the local Content definitions. The browser matches LCD surface material names from the active surface to parsed MWM material groups and replaces those materials with either the selected local LCD image texture or a generated canvas texture for text/sprite/background composition. LCD replacement geometry stays depth-tested with the rest of the model, matching the game's material-texture swap behavior; when a non-transparent LCD surface is replaced, matching `ScreenArea*` base material groups are omitted to avoid coplanar z-fighting with nearby screen backing geometry. Static LCD images load from the selected local `Content` folder; text content, font/color settings, and programmable-block sprite text layers are sent as metadata so the browser can render characters without receiving the game's offscreen render target. Regular LCD `BackgroundAlpha` is treated as panel metalness rather than canvas opacity; transparent LCDs keep their active screen translucent from `TransparentMaterials.sbc`, use selected non-black background colors even when the saved `BackgroundAlpha` is zero, and leave inactive `TransparentScreenArea*` rotation planes on their default transparent material without block paint tint. Script-generated texture sprites use their logical LCD texture paths and are loaded locally when present. Dedicated-server builds do not recreate built-in client LCD scripts pixel-for-pixel, so surfaces whose visible output exists only as a client-side generated texture may fall back to their background/text metadata.

Transparent LCD replacement surfaces are rendered single-sided, matching the game's transparent material path. This prevents the front and back transparent screen planes from drawing the same LCD output over each other while preserving the inactive rotation planes' default transparent material. Transparent LCD batches also render in the same late transparent layer as glass/holo/shield materials, after the floor grid. This keeps the floor grid consistently tinted by the screen instead of letting Three.js depth-sort the grid and LCD against each other differently as the camera angle changes.

DDS textures used inside LCD canvas composition are decoded through WebGL readback before Canvas2D draws them, because compressed Three.js textures are not browser image objects. The LCD canvas path also detects sprite-atlas images whose useful RGB content has a near-zero alpha plane and restores drawable alpha from RGB, matching Space Engineers' premultiplied sprite-atlas rendering used by built-in wide placeholders such as `Screen Online` wide.

Empty and unpowered LCD screens follow Space Engineers' built-in placeholder rules. Powered `ContentType.NONE` surfaces show `Screen Online` or `Screen Online` wide only when the game-side `UseOnlineTexture` flag is enabled; otherwise the viewer follows the game's reset-to-model branch for the screen area. Unpowered surfaces use `Screen Offline` or the wide offline placeholder. Blocks that define `MaterialNamesToHideWhenOffline` also omit those model material groups while the block's LCD surfaces are offline, matching holo, transparent, and special screen backings that disappear in game when power is lost.

For powered `ContentType.NONE` surfaces with `UseOnlineTexture` disabled, Space Engineers clears the runtime material override and removes the non-transparent screen-area render flag. The viewer mirrors that by omitting the matching screen material group, revealing the block's built-in/default screen geometry instead of drawing the model's black LCD placeholder texture over it.

LCD text uses the real Space Engineers bitmap font data for every built-in font exposed by the LCD font menu: `Debug`, `Red`, `Green`, `Blue`, `White`, `DarkBlue`, `UrlNormal`, `UrlHighlight`, `ErrorMessageBoxCaption`, `ErrorMessageBoxText`, `InfoMessageBoxCaption`, `InfoMessageBoxText`, `ScreenCaption`, `GameCredits`, `LoadingScreen`, `BuildInfo`, `BuildInfoHighlight`, and `Monospace`. The browser maps those definitions to the same first-pass atlases and color masks used by `Fonts.sbc`: shadowed definitions use `Fonts/white_shadow/FontDataPA.xml`, white-family definitions use `Fonts/white/FontDataPA.xml`, and `Monospace` uses `Fonts/monospace/FontDataPA.xml`. The atlas pages are loaded from the selected local `Content` folder, decoded once through WebGL readback, unpremultiplied for `FontDataPA` before Canvas2D tinting, and drawn with the game's advance, bearing draw offset, line-height, per-font color mask, and `144/185` GUI text scale. LCD canvas composition uses the game's `SurfaceSize` layout coordinates and then maps that result into the raw render texture, matching non-square screen aspect handling on curved and round LCD panels. Unknown LCD font names still fall back to the `Debug` definition. When the viewer has to use its Canvas2D fallback text path before a bitmap font is available, it preserves only explicit newline breaks and does not soft-wrap overflowing text, matching the game's clipped overflow behavior more closely.

Applied block paint is rendered client-side from the scene `colourMaskHsv` metadata. Instanced model batches carry paint through per-instance color data instead of allocating a repeated per-vertex paint buffer for every rendered block. For textured models, the viewer uses Space Engineers-style color masking: base color comes from `ColorMetalTexture`/diffuse textures, paint strength comes from `AddMapsTexture`/extension-map alpha, and `*_alphamask.dds` is treated only as alpha/cutout data rather than a paint mask. Parsed model materials only receive block paint when their resolved material texture set includes an extension/add-map paint mask. Materials without a paint mask, such as the vanilla camera `CameraLens` material, keep their authored texture color instead of using heuristic fallback tinting. When the selected local Content folder contains `Data/AssetModifiers/ArmorModifiers.sbc`, the browser resolves block `skinSubtypeId` values to the same material/location texture overrides used by the game client and swaps `ColorMetal`, `NormalGloss`, `Extensions`, and `Alphamask` paths before loading material textures. Alpha-masked model techniques are rendered as depth-writing cutouts instead of blended transparent sheets, and decal techniques use a small WebGL polygon depth offset to match the game's static-decal depth bias and avoid coplanar panel/detail z-fighting. Glass/holo/shield techniques render as blended materials with depth writes disabled to reduce self-overlap artifacts. `GLASS` mesh parts and named transparent screen-area materials use transparent-material texture/color data from `TransparentMaterials.sbc` and do not consume the block paint mask, so painted frames can still be colored while transparent glass and transparent LCD planes keep their normal color. Glass uses a viewer-side approximation of Space Engineers' transparent material constants: `Color.W` is not used as raw Three.js opacity, texture alpha is remapped to avoid a second opacity loss, and `ColorAdd`, light/shadow multipliers, reflectivity, fresnel, gloss, and gloss-texture add values are folded into the browser material/shader setup. One-way window glass treats `*Outside` transparent materials as the darker mirrored side and `*Inside` materials as the clearer side, matching the game's use of low `Color.W` plus high reflectivity/additive constants on outside glass. When a local model cannot be parsed, proxy boxes use the same block paint color as a visual fallback.

The viewer statistics panel reports unique model assets listed by the scene, model files found locally, parsed models, missing models, active scene mod count, context mode/grid/clipping/block counters, LCD surface count, grid light source counts, voxel mesh counts, and the equivalent texture counts discovered from scene texture metadata and locally parsed MWM material groups. Texture counts are unique by source root plus logical texture path; `loaded` counts textures the browser successfully decoded/uploaded, while `failed` covers locally found textures that could not be used, such as unsupported DDS/WebGL compression combinations. Texture found/missing/loaded/failed counts update progressively as visible/shared materials request textures.

The stats panel exposes timing counters for the main asset pipeline: scene snapshot fetch, model path resolution, local file metadata reads, MWM file reads, MWM parse time, total model resolution, texture path resolution, DDS file reads, DDS parse time, and compressed texture upload validation/init. Path resolution covers handle lookup only; local file metadata reads cover `getFile()` snapshots; byte-read timings cover `arrayBuffer()` work. Successful DDS loads are written to the browser console at debug level instead of rebuilding the visible warning log for every texture; the visible log batches DOM updates and is primarily for warnings/fallbacks.

Loaded voxel bodies can be shown as textured terrain mesh when the viewer URL includes `voxels=1`. The agent sends bounded voxel content/material chunks that intersect the same snapped X/Z footprint covered by the viewer floor grid, with matching two-major-square padding and additional vertical padding, and stops at conservative safety limits. When the whole bounded range fits the payload budget, it is sent as one range so terrain surfaces crossing internal chunk boundaries are preserved; larger ranges fall back to mixed chunk sampling and skip uniform empty/full chunks. The browser procedurally builds the terrain surface from those density samples, clips generated voxel polygons to the same snapped X/Z bounds used by the floor grid, groups triangles by material index, projects UVs from each generated polygon's dominant axis, and resolves the sampled voxel material `ColorMetal` and `NormalGloss` texture paths from the selected local `Content` folder. Generated voxel terrain writes normals from the scalar-field gradient and orients generated polygons to match those normals before indexing, so the normal-facing side is the shadow-casting side instead of alternating across tetrahedral mesh triangles. Voxel terrain is rendered visually double-sided for caves and overhangs without forcing separate reverse-face shadow rendering, matching Space Engineers' normal-facing voxel lighting convention while preserving shadows from hills, caves, and other relief. Voxel `ColorMetal` alpha drives metalness and `NormalGloss` alpha drives gloss-to-roughness conversion through the same Space Engineers shader approximation used for block materials. Missing or unloadable local voxel textures fall back to deterministic generated colors. If data chunks arrive but generate no triangles, the viewer logs chunk size, byte counts, content range, iso-crossing status, and voxel-metadata availability to aid diagnosis.

Voxel mesh generation follows the game's coordinate convention: storage coordinates are first converted to absolute world positions with `PositionLeftBottomCorner + voxelCoord`, then transformed into the viewer's grid-relative coordinate space. The voxel entity `WorldMatrix` translation is the body center and must not be used as the storage-coordinate origin.

For non-context grid scenes, the browser initializes voxel clipping from the selected grid's relative block footprint before terrain meshing. That keeps the procedural voxel polygons clipped to the same snapped X/Z footprint later used by the visible floor grid.

Grid scenes automatically choose the lowest voxel LOD that fits the current intersecting voxel ranges within the scene chunk and byte budgets before sampling. The browser honors the chunk `lod` scale the same way as standalone asteroid scenes, so coarser fallback samples keep the correct world-space size instead of being skipped when LOD 0 would exceed the payload limits.

When both `voxels=1` and `context=1` are enabled for a grid, voxel body metadata and voxel data chunks are bounded to the context volume instead of only the selected grid footprint. The server samples voxels from the enclosing world AABB of the selected-grid-local context volume, while the browser clips generated terrain to the matching selected-grid-relative context bounds. The same metadata-only asset boundary still applies: Quasar sends sampled live voxel scalar/material data and logical voxel material paths, not Space Engineers asset bytes.

Voxel terrain UVs follow the Space Engineers voxel material tiling convention by using the inverse of each material's `TilingScale`, falling back to the legacy `1/8` scale only when metadata is unavailable or invalid. Generated voxel mesh groups are also split by dominant-axis projection so top/bottom-like surfaces prefer `ColorMetalY` and `NormalGlossY`, while side-like surfaces prefer `ColorMetalXZnY` and `NormalGlossXZnY`; both paths fall back to the alternate axis texture when needed.

Grid scenes keep camera, floor-grid, and sun bounds anchored to rendered grid-group bounds rather than voxel meshes, full voxel-body proxies, or transformed world AABBs, so large intersecting voxel bodies do not pull the initial camera or floor grid away from the selected grid.

Voxel-only asteroid scenes use rendered voxel-group bounds for camera, floor-grid, and sun placement so the sampled asteroid mesh remains centered even when metadata bounds and generated terrain bounds differ.

Standalone asteroid scenes use the same live voxel storage sampling path with separate chunk and byte limits for the selected asteroid body. They automatically choose the lowest voxel LOD that fits those limits, skip uniform empty/full chunks, center the relative browser view on the asteroid `WorldAABB`, and disable floor-footprint clipping so the whole sampled asteroid mesh can render. The browser honors each voxel data chunk's `lod` value by scaling sample coordinates by `2^lod`, so lower-detail asteroid samples still occupy the correct world-space size. Asteroid floor grids use 2.5 m minor squares and 12.5 m major lines, preserving the same 5x visual scale jump from large grids to asteroid scenes as from small grids to large grids. Fog density scales down from the active floor-grid span so terrain remains visible across several complete floor grids before reaching the obscured threshold. Planets remain unsupported in this pass.

The `Show voxels` control is disabled unless the URL includes a `voxels` parameter. `voxels=1`, `voxels=true`, `voxels=yes`, and an empty `voxels` value enable and check it by default. `voxels=0`, `voxels=false`, and `voxels=no` leave it enabled but unchecked and request the initial scene without voxel data. If a voxel body has no data chunk because it was skipped, capped, or failed, the viewer may still show the old metadata proxy as a fallback: planets use a wire sphere from radius metadata, and other voxel maps use their world bounds.

The `Show lighting` toggle controls all explicit viewer lights. When enabled, the sun position comes from `MySector.DirectionToSunNormalized` in the running game and is rendered as a directional light with the existing marker and ray helper. The renderer enables Three.js shadow maps, and the sun casts shadows from model and proxy grid geometry through a tightly fitted orthographic shadow camera with texel-scaled normal bias instead of a point-light cube shadow; voxel terrain participates in the same shadow-map path as generated browser geometry. Projected/reflector spot lights can also cast shadows under a four-light cap that follows Space Engineers' active spot-shadow budget model, while point/interior lights and reflector companion fill lights do not cast shadows. Captured in-game grid lights are also rendered from metadata-only light records, including terminal-synced color, radius, falloff, intensity, reflector cone data, blink settings, and server working state. The agent includes classic block lights, component-based lights, and runtime-subpart lights exposed through Space Engineers lighting logic; subpart records are folded into the scene light list and also remain attached to their subpart metadata for consumers that inspect block geometry. The browser de-duplicates top-level and subpart light records before rendering or reporting stats. The browser maps Space Engineers falloff into Three.js decay with the direction inverted, so lower in-game falloff values keep lights tighter instead of expanding their viewer influence. The agent does not use `CurrentLightPower` for snapshots because that value is a client/render fade derived from working state and can remain zero on dedicated servers. Interior/component lights become point lights; reflector blocks become spotlights with a weak companion point light for local fill. To protect browser performance, the viewer instantiates a deterministic capped subset of the highest-impact grid lights while still reporting the captured and capped counts in the stats panel. When disabled, all explicit sun and grid lights are hidden and the viewer uses stronger diffuse-only ambient lighting so the grid remains readable without scene lights. The fallback does not use an environment map, so ambient illumination does not add point-like specular highlights or glass reflections when the sun is obscured.

The stats panel also includes live WebGL and viewport counters such as draw calls, triangles, lines, points, GPU geometries/textures, shader programs, renderables, visible objects, culled objects, meshes, sprites, and lights.

Missing or unparseable local models and missing or unloadable textures are non-fatal. The viewer logs warnings and keeps the scene visible with proxy boxes and generated fallback materials where needed. Local Content and Mods lookups are cached in memory for the current selected folders, including resolved paths, misses, in-flight full-path lookups, typed directory/file child lookups, and case-insensitive directory entry maps. Intermediate path segments only probe directories and final path segments only probe files, avoiding slow wrong-kind probes such as treating `.mwm` filenames as directories. `getFile()` metadata snapshots are deferred until size, mtime, or bytes are needed and are cached separately by root and canonical path. `.sbm` and legacy `*_legacy.bin` archive indexes are also tied to the current asset cache generation. Once a directory has been enumerated, later child lookups use the cached lowercase map first to avoid repeated expensive failed exact-case probes. Firefox's folder-input fallback feeds the same cache through virtual directory/file handles backed by the selected `File` objects. Cache diagnostics in the stats panel include path hits/misses, exact probes, enumerations, case-fallback hits, negative-cache hits, and metadata-cache hits. The caches are cleared when a different Content or Mods folder is selected or restored.

## Server-Side Notes

The scene snapshot is captured by `Quasar.Agent` on the game thread through the existing agent command/result WebSocket flow. The command is `GetEntityRenderScene`, and the shared DTOs live in `Magnetar.Protocol`.

The dedicated-server agent deliberately does not reference client render assemblies to resolve skin texture-change payloads. It sends `SkinSubtypeId` and other block metadata; local browser-side content handling resolves base-game armor skin texture overrides from the selected Content folder.
